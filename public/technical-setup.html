Setup steps — in order
1
Supabase — Auth + Database
Creates the user system and replaces localStorage. Everything else depends on this.
Create Supabase project
supabase.com → New project
  • Create project, note the project URL and anon key
  • Enable Email + Google OAuth providers
  • Create the tables below (schema section)
  • Enable Row Level Security on every table
  • Add RLS policy: user_id = auth.uid() on all rows
Wire into frontend
npm install @supabase/supabase-js
  • Create src/lib/supabase.js with client init
  • Replace AppContext.jsx localStorage reads/writes with Supabase queries
  • Add login/logout flow (currently no auth in demo)
  • On login: load user's voiceGuide, posts, assets from DB
  • On state change: write to DB instead of localStorage
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbG...
2
Generation API — Move AI off the browser
Claude API key currently lives in the browser (demo only). Move it server-side before any real users.
Create serverless function
Vercel function / Supabase Edge Function
  • Create api/generate.js (or edge function)
  • Move all logic from src/lib/anthropic.js into it
  • Add per-user quota check against Supabase before calling Claude
  • Log token usage to generation_log table after each call
  • Return post objects to frontend — same shape as current
Update frontend call
src/lib/anthropic.js → update fetch target
  • Change fetch URL from api.anthropic.com to /api/generate
  • Remove anthropic-dangerous-direct-browser-access header
  • Pass Supabase auth token in Authorization header
  • Server verifies token, checks quota, calls Claude
  • scrapeAndFillVoiceGuide() stays browser-side (Jina Reader is public)
# Server-side only — never in frontend
ANTHROPIC_API_KEY=sk-ant-...
GOOGLE_AI_API_KEY=AIza... # Nano Banana + Veo 3
Nano Banana + Veo 3: Add Google AI Studio / Vertex AI calls to the same serverless function. Claude generates visualBrief and referenceAssets → pass those to Nano Banana (images) and Veo 3 (video/reel) in parallel. Store generated media in Cloudflare R2, attach CDN URL to post object.
3
Cloudflare R2 — Asset & Media Storage
Replace base64 dataURLs in localStorage with real CDN-hosted files.
Brand assets (user-uploaded)
R2 bucket: act-brand-assets
  • Create R2 bucket with public access
  • Update src/pages/Assets.jsx — on upload, POST file to /api/upload-asset
  • Server uploads to R2, returns CDN URL
  • Store CDN URL in Supabase assets table (not base64)
  • Asset manifest sent to Claude = labels + CDN URLs
AI-generated media
R2 bucket: act-generated-media
  • After Nano Banana generates image → upload to R2
  • After Veo 3 generates video → upload to R2
  • Store CDN URL in Supabase posts.media_url
  • Swipe card renders from CDN URL not gradient placeholder
  • CDN URL sent to GHL Social Planner with the post

4
GoHighLevel — OAuth + Social Planner
Wire GHL_INTEGRATION.pushToGHL() in src/lib/anthropic.js. This is where posts go after approval.
GHL OAuth Setup
marketplace.gohighlevel.com → Create App
  • Create a Marketplace app, get Client ID + Secret
  • Set OAuth redirect URI to /api/ghl/callback
  • Scopes needed: social-media-posting.write, contacts.write, opportunities.write
  • Build connect flow: user clicks "Connect GHL" → OAuth → store access_token + refresh_token + locationId in Supabase
  • Build token refresh cron (GHL tokens expire)
Social Planner — Post Push
POST services.leadconnectorhq.com/social-media-posting
  • On post approval → call /api/ghl/schedule-post
  • Map ACT post → GHL payload: caption, platforms array, scheduledAt (UTC), mediaUrls
  • scheduledDay + scheduledTime + user timezone → real UTC datetime
  • Handle per-platform char limits (X = 280, etc.)
  • GHL returns post ID → store in Supabase posts.ghl_post_id
GHL_CLIENT_ID=...
GHL_CLIENT_SECRET=...
# Per-user tokens stored in Supabase, not env
Integration point in codebase: Open src/lib/anthropic.js → find GHL_INTEGRATION.pushToGHL() at the bottom. Replace the throw new Error with a fetch('/api/ghl/schedule-post', ...) call. The post object shape and ghlConfig are already passed in correctly.
5
GHL Workflows — Automation Layer
Build these workflows in GHL. ACT fires webhooks to trigger them — no custom notification system needed.
user.created
GHL: Create Contact → assign to "New User" pipeline stage → send welcome email → trigger intro call booking via Calendar
voiceguide.completed
GHL: Update Contact field voiceGuideComplete = true → move pipeline to "Guide Built" → send "You're set up" email
posts.generated
GHL: Send SMS/email "Your posts are ready to review" → link to /swipe
post.approved
GHL: Move pipeline to "Actively Posting" → send confirmation with scheduled date/platform
post.published
GHL: Send "Your post is live" notification → log activity on Contact → trigger engagement check-in after 48h
quota.warning
GHL: Send "You have X generations remaining this week" warning → prompt to upgrade if on Starter plan
user.inactive (7 days)
GHL: Move pipeline to "At Risk" → fire re-engagement sequence → flag account manager
Build a single /api/ghl/webhook-out endpoint in your backend. ACT calls it on each state change. The endpoint maps the event type to a GHL workflow trigger via the GHL Workflow API. All notification logic lives in GHL — nothing to build custom.
6
GHL CRM — Contacts + Pipeline
Every ACT user = a GHL Contact. Pipeline stages move automatically from the webhook bridge.
Contact custom fields
GHL → Settings → Custom Fields → Contact
  • brandName (text)
  • voiceGuideComplete (checkbox)
  • generationsUsedThisWeek (number)
  • planTier (dropdown: Starter/Pro/Agency)
  • activePlatforms (multi-select)
  • actUserId (text) — links GHL Contact to Supabase user
Pipeline stages
GHL → Pipelines → Create "ACT Clients"
  • Signed Up
  • Voice Guide Built
  • First Posts Generated
  • First Post Approved
  • Actively Posting Weekly
  • At Risk (missed a week)
  • Churned
7
Stripe — Subscription Billing
Quota enforcement moves server-side. Plan tier controls generation limit.
Products + prices
dashboard.stripe.com → Products
  • Starter — 20 generations/week
  • Pro — 60 generations/week
  • Agency — unlimited generations
  • Set up monthly + annual prices for each
Webhook → plan sync
/api/stripe/webhook
  • checkout.session.completed → update user plan in Supabase
  • customer.subscription.updated → update plan tier
  • customer.subscription.deleted → downgrade to Starter
  • Generation quota checked server-side against plan tier before every API call

STRIPE_WEBHOOK_SECRET=whsec_...
8
Infrastructure — CI/CD + Monitoring
Set this up before going live with real users.
Deployment
  • Connect repo to Vercel
  • Set all env vars in Vercel dashboard
  • Create Preview + Production environments
  • Serverless functions deploy automatically
Error tracking
  • Install Sentry: npm install @sentry/react
  • Add Sentry.init() in src/main.jsx
  • Add Sentry to serverless functions too
  • Set up Slack alert for new issues
Rate limiting
  • Add Upstash Redis for rate limiting
  • Limit /api/generate: 10 req/min per user
  • Limit /api/ghl/schedule-post: 20 req/min
  • Return 429 with clear error message
